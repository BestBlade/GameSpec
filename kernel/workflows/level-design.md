# 关卡设计工作流

## 概述

本工作流覆盖从关卡概念到最终验证的完整关卡设计路径。通过关卡概念、空间语法与节奏、实体编排与叙事、验证与审查四个阶段，确保每个关卡在空间设计、战斗节奏、叙事融合与数值平衡上均达到设计标准。

## P1/P2 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L2 角色新建或修改项目文档后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可交接、提审或进入下一阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若当前活跃文档发生切换，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 关卡概念

| 属性 | 值 |
|------|-----|
| **层级** | L0 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-关卡设计（主导）、@game-玩法策划（玩法约束）、@game-导航（启动分发） |

### 输入

- 关卡需求描述（位于游戏流程中的位置、目标体验）
- 核心玩法定义
- 关联叙事章节（如有）

### 工作内容

1. **关卡目标定义** — 明确关卡的设计目标（教学/挑战/叙事推进/探索奖励）
2. **体验目标设定** — 定义玩家在该关卡中的目标情感体验曲线
3. **核心挑战定义** — 确定关卡的核心挑战类型（战斗/解谜/平台/混合）
4. **玩法约束确认** — @game-玩法策划 确认该关卡可使用的机制、能力、资源

### 输出

- 关卡概念文档（目标、体验、挑战类型、约束）

### 质量门禁

- 关卡目标与所在流程位置匹配
- 体验目标有情感曲线描述
- 核心挑战与可用玩法机制匹配

## Phase 2: 空间语法与节奏

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-关卡设计（主导）、@game-数值（难度曲线） |
| **所需 Skill** | `level-spec`、`mermaid-drawing` |

### 输入

- 关卡概念文档（Phase 1 产出）

### 工作内容

1. **空间拓扑设计** — 调用 `mermaid-drawing` 绘制关卡空间拓扑图（房间/区域连接关系、关键路径与支线路径）
2. **节奏曲线设计** — 设计关卡内的紧张-放松节奏曲线，标注高潮点与休息点
3. **难度曲线** — @game-数值 设计关卡内的难度递进曲线，与节奏曲线对齐
4. **空间叙事标注** — 在空间拓扑中标注环境叙事元素的预设位置
5. **关卡规范产出** — 调用 `level-spec` 技能，按标准格式输出关卡设计文档

### 输出

- 空间拓扑图
- 节奏 & 难度曲线图
- 关卡设计文档（初稿）

### 质量门禁

- 空间拓扑图包含主路径与至少 1 条支线路径
- 节奏曲线有明确的高潮点定义
- 难度曲线与节奏曲线节点对齐

## Phase 3: 实体编排与叙事

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-关卡设计（主导）、@game-叙事（叙事融合）、@game-数值（实体数值） |
| **所需 Skill** | `level-spec` |

### 输入

- 关卡设计文档初稿（Phase 2 产出）
- 叙事节拍表（来自叙事设计工作流，如有）

### 工作内容

1. **敌人/实体编排** — 在空间拓扑中配置敌人、陷阱、可交互物件的类型与数量
2. **实体数值配置** — @game-数值 为编排的实体配置具体数值（HP、伤害、掉落等）
3. **叙事融合** — @game-叙事 将叙事节拍嵌入关卡流程，设计环境叙事与事件触发点
4. **引导设计** — 设计玩家引导元素（光照引导、声音提示、NPC引导对白）
5. **关卡文档更新** — 调用 `level-spec` 更新关卡设计文档至完整版

### 输出

- 完整关卡设计文档（含实体编排、数值、叙事、引导）

### 质量门禁

- 实体编排与难度曲线一致
- 叙事节拍与关卡节奏不冲突
- 引导设计覆盖关键决策点

## Phase 4: 验证与审查

| 属性 | 值 |
|------|-----|
| **层级** | L3 |
| **周期** | 2-3 天 |
| **参与 Agent** | `workflow: document-review`（`@game-规范审查` → `@game-逻辑验证` → `@game-Spec架构师`） |
| **所需 Skill** | `document-validator`、`logic-review`、`spec-standard-enforcer`、`structure-diff`、`review-scoring` |

### 输入

- 完整关卡设计文档（Phase 3 产出）
- 评分标准文档（如无，则使用项目默认评分标准）

### 审查模式适用

- 本阶段接入 `workflow: document-review`，其 `review_mode` 分发规则以该工作流定义为准。
- 进入验证与审查的正式提交默认按 `full` 处理。
- `lean` 可用于局部补写或单关卡修订的低成本复核，但若未走完完整回路，不得直接定稿。
- `prototype` 仅适用于探索性关卡稿的最小结构检查，不得直接作为本阶段定稿输入。

### 工作内容

1. **规范审查** — 先检查文档格式、命名和链接规范，阻断结构性问题，并识别 `review_mode`
2. **逻辑验证** — 按 `review_mode` 分发规则检查关卡流程逻辑、触发器、可达性和关键路径
3. **Spec 架构复核** — 按 `review_mode` 分发规则检查模板完整性、跨文档引用与关卡包结构质量
4. **可玩性评估与统一评分** — 基于文档推演玩家体验路径，结合 `review-scoring` 给出总分与维度分
5. **问题修复与定稿** — 根据统一审查结论回流修订或进入定稿

### 输出

- 规范审查报告
- 逻辑验证报告
- Spec 架构复核报告
- 关卡评审评分报告
- 关卡设计文档（定稿）

### 质量门禁

- 无死路、无不可达区域
- 触发器逻辑零矛盾
- 文档格式合规率 100%
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 或存在阻塞问题不得定稿；6.0 - 6.9 需修订后复审；7.0+ 方可直接定稿
- 所有阻塞问题清零方可定稿
