# 数值设计工作流

## 概述

本工作流覆盖从系统需求中的数值部分到最终可交付配置表的完整路径。通过数值建模、模拟验证、平衡调优、配置生成、数值评审五个阶段，确保游戏数值体系的科学性、平衡性与可维护性。

## P1/P2 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L2 角色新建或修改项目文档、配置说明文档或项目根 `active.md` 后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可交接、提审或进入下一阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若当前活跃文档发生切换，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 数值建模

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-数值（主导）、@game-系统策划（需求对齐）、@game-玩法策划（体验目标） |
| **所需 Skill** | `economy-modeling` |

### 输入

- 系统设计文档（含数值相关需求）
- 核心循环定义
- 设计目标与体验锚点

### 工作内容

1. **数值需求拆解** — @game-数值 从系统设计文档中提取所有数值相关需求，分类归纳
2. **经济模型构建** — 调用 `economy-modeling` 技能，构建资源产出/消耗模型，定义经济循环
3. **成长曲线设计** — 设计角色/系统成长曲线，定义关键节点与拐点
4. **公式体系建立** — 建立核心计算公式（伤害、收益、概率等），标注各参数含义与取值范围

### 输出

- 数值模型文档（公式集、参数表、成长曲线）
- 经济模型图

### 质量门禁

- 所有公式参数有明确定义与合理取值范围
- 经济模型无资源黑洞或无限产出漏洞
- 成长曲线关键节点与设计目标一致

## Phase 2: 模拟验证

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-数值（主导） |
| **所需 Skill** | `numerical-simulation`、`exploit-testing` |

### 输入

- 数值模型文档（Phase 1 产出）

### 工作内容

1. **模拟场景定义** — 定义典型玩家行为场景（高活跃/低活跃/极端操作）
2. **数值模拟运行** — 调用 `numerical-simulation` 技能，对各场景进行蒙特卡洛模拟或确定性推演
3. **漏洞检测** — 调用 `exploit-testing` 技能，检测是否存在可被利用的数值漏洞（刷资源、无限循环等）
4. **模拟结果分析** — 分析模拟数据，标注异常值与风险区域

### 输出

- 模拟验证报告（含各场景数据与分析）
- 漏洞检测报告

### 质量门禁

- 模拟场景覆盖至少 3 类玩家行为模式
- 无高危数值漏洞
- 模拟结果在设计预期范围内（偏差 ≤ 15%）

## Phase 3: 平衡调优

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 2-4 天 |
| **参与 Agent** | @game-数值（主导）、@game-玩法策划（体验反馈） |
| **所需 Skill** | `numerical-simulation` |

### 输入

- 模拟验证报告（Phase 2 产出）
- 漏洞检测报告

### 工作内容

1. **问题定位** — 根据模拟报告中的异常值与风险区域，定位需要调整的参数
2. **参数调优** — 调整公式参数或结构，修复检测到的漏洞
3. **二次模拟验证** — 调用 `numerical-simulation` 重新模拟，确认调优效果
4. **体验校准** — @game-玩法策划 从玩家体验角度评估数值手感，提出调整建议
5. **迭代收敛** — 重复调优-模拟循环直至数据稳定且体验达标

### 输出

- 调优后的数值模型（终版）
- 平衡性验证报告

### 质量门禁

- 所有 Phase 2 发现的漏洞已修复
- 二次模拟偏差 ≤ 10%
- 玩法策划确认体验手感达标

## Phase 4: 配置生成

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 1-2 天 |
| **参与 Agent** | @game-数值（主导）、@game-系统策划（配置结构确认） |
| **所需 Skill** | `config-table-gen` |

### 输入

- 调优后的数值模型（Phase 3 产出）

### 工作内容

1. **配置表结构设计** — @game-系统策划 确认配置表结构（字段名、类型、约束）
2. **配置数据生成** — 调用 `config-table-gen` 技能，基于终版数值模型批量生成配置表数据
3. **数据校验** — 校验生成数据的完整性、范围合法性、引用一致性
4. **配置文档编写** — 编写配置表使用说明，标注每个字段的业务含义与修改注意事项

### 输出

- 配置表文件（结构化格式）
- 配置表使用说明文档

### 质量门禁

- 配置表字段 100% 有类型定义与取值范围
- 数据无空值、越界、引用断裂
- 使用说明覆盖所有可调参数

## Phase 5: 数值评审

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 2 天 |
| **参与 Agent** | @game-技术总监（技术可行性）、@game-导航（流程管理） |
| **所需 Skill** | `review-scoring` |

### 输入

- 数值模型文档（终版）
- 配置表与使用说明
- 模拟验证报告 & 平衡性验证报告
- 评分标准文档（如无，则使用项目默认评分标准）

### 工作内容

1. **技术可行性确认** — @game-技术总监 确认配置表结构可被开发直接使用，无技术障碍
2. **整体评审** — 评审数值体系的完整性、合理性、可维护性
3. **统一评分** — 调用 `review-scoring`，基于模型完整度、平衡性、可维护性、可交付性给出总分与维度分
4. **评审决策** — L1 审批：通过 / 有条件通过 / 打回

### 输出

- 数值评审报告
- 数值评审评分报告
- 评审结论（通过 / 有条件通过 / 打回至对应阶段）

### 质量门禁

- L1 审批通过方可交付开发
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 直接打回；6.0 - 6.9 最多有条件通过；7.0+ 方可直接通过
- 打回时需指明打回至哪个阶段及修改要求
- 通过后的配置表锁定版本号
