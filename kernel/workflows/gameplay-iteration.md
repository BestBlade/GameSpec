# 核心玩法迭代工作流

## 概述

本工作流定义了核心玩法的快速闭环迭代流程。每一轮迭代从体验锚点出发，经过循环设计、数值与技术验证、逻辑验证，形成完整的迭代闭环。适用于原型阶段和正式开发阶段的玩法打磨。

## P1/P2/P3 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L2 角色新建或修改项目文档后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可交接、提审或进入下一阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若本轮迭代的问题已收敛到 UE5 的实现边界、运行时状态、SaveGame 或性能预算细节，可在技术验证阶段按需调用 `@game-UE技术顾问` 输出 `UE 可落地性评估`；方向性技术取舍仍先由 `@game-技术总监` 判断。
- 若当前活跃文档发生切换，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## Phase 1: 体验锚定

| 属性 | 值 |
|------|-----|
| **层级** | L0 |
| **周期** | 1-2 天 |
| **参与 Agent** | @game-玩法策划（主导）、@game-导航（启动分发） |

### 输入

- 核心玩法定义（首轮）或上一轮迭代反馈（后续轮次）
- 设计支柱文档
- 玩家体验目标

### 工作内容

1. **体验锚点定义** — @game-玩法策划 明确本轮迭代的核心体验锚点（"玩家在这个瞬间应该感受到什么"）
2. **迭代范围圈定** — 明确本轮迭代的聚焦范围，排除不在本轮处理的内容
3. **成功标准设定** — 定义本轮迭代的可衡量成功标准
4. **上轮问题复盘** — 如非首轮，复盘上轮发现的问题与改进方向

### 输出

- 体验锚点定义
- 迭代范围与成功标准
- 问题复盘记录（非首轮）

### 质量门禁

- 体验锚点可感知、可描述
- 迭代范围聚焦，不超过 3 个核心问题
- 成功标准可测量

## Phase 2: 循环设计

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-玩法策划（主导）、@game-数值（数值支撑） |
| **所需 Skill** | `gameplay-spec`、`mermaid-drawing` |

### 输入

- 体验锚点定义（Phase 1 产出）
- 现有玩法设计文档（如有）

### 工作内容

1. **循环结构（再）设计** — 调用 `gameplay-spec` 技能，设计或调整玩法循环结构（输入→处理→反馈→奖励）
2. **流程图绘制** — 调用 `mermaid-drawing` 绘制玩法流程图、状态转换图
3. **反馈系统设计** — 设计正负反馈机制，确保体验锚点能被传达
4. **数值框架** — @game-数值 为本轮设计提供数值框架（参数范围、曲线类型）
5. **边界情况处理** — 定义异常操作、极端情况下的玩法行为

### 输出

- 玩法循环设计文档
- 玩法流程图 & 状态转换图
- 数值框架文档

### 质量门禁

- 循环结构闭环完整（无断路）
- 反馈机制与体验锚点匹配
- 边界情况有明确处理策略

## Phase 3: 数值与技术验证

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-技术总监（技术验证）、@game-数值（数值验证） |
| **所需 Skill** | `numerical-simulation`、`exploit-testing`、`review-scoring` |

### 输入

- 玩法循环设计文档（Phase 2 产出）
- 数值框架文档
- 评分标准文档（如无，则使用项目默认评分标准）

### 工作内容

1. **数值模拟** — @game-数值 调用 `numerical-simulation` 对玩法核心数值进行模拟验证
2. **漏洞检测** — 调用 `exploit-testing` 检测玩法漏洞（无限循环、零成本获利等）
3. **技术可行性确认** — @game-技术总监 评估本轮设计的技术实现可行性与性能影响
4. **统一评分** — 调用 `review-scoring`，基于体验锚点达成度、循环闭环质量、数值稳定性、技术可行性给出总分与维度分
5. **验证结论** — 汇总数值与技术验证结果，给出通过/调整/打回决策

### 输出

- 数值模拟报告
- 漏洞检测报告
- 技术可行性评估
- 本轮迭代评分报告
- 验证结论（通过 / 需调整 / 打回 Phase 2）

### 质量门禁

- 数值模拟结果在预期范围内
- 无高危玩法漏洞
- 技术可行性无阻塞性问题
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 打回 Phase 2；6.0 - 6.9 进入调整；7.0+ 方可继续
- L1 审批通过方可继续

## Phase 4: 文档复核与迭代闭环

| 属性 | 值 |
|------|-----|
| **层级** | L3 |
| **周期** | 1-2 天 |
| **参与 Agent** | `workflow: document-review`（`@game-规范审查` → `@game-逻辑验证` → `@game-Spec架构师`） |
| **所需 Skill** | `document-validator`、`logic-review`、`spec-standard-enforcer`、`structure-diff`、`review-scoring` |

### 输入

- 通过 Phase 3 验证的玩法设计文档
- 数值模拟报告

### 审查模式适用

- 本阶段接入 `workflow: document-review`，其 `review_mode` 分发规则以该工作流定义为准。
- 进入迭代闭环确认的正式提交默认按 `full` 处理。
- `lean` 可用于单轮局部修订的低成本复核，但若未走完完整回路，不得把本轮迭代标记为正式闭环完成。
- `prototype` 仅适用于探索性玩法稿的最小结构检查，不得直接作为本阶段闭环确认输入。

### 工作内容

1. **规范审查** — 先检查玩法文档结构、命名和变量表达是否合规，并识别 `review_mode`
2. **逻辑验证** — 按 `review_mode` 分发规则检查玩法规则、状态空间和极端场景的逻辑闭合度
3. **Spec 架构复核** — 按 `review_mode` 分发规则检查 MDA 结构完整性、模板覆盖与跨文档引用质量
4. **迭代闭环确认** — 结合三层审查结果，确认本轮迭代目标达成情况并记录遗留问题

### 输出

- 规范审查报告
- 逻辑验证报告
- Spec 架构复核报告
- 迭代闭环总结（目标达成 / 遗留问题清单）
- 下轮迭代建议

### 质量门禁

- 规则逻辑零矛盾
- 状态机无死锁
- 迭代闭环总结包含明确的达成/未达成判定
