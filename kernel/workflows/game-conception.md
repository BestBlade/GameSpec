# 游戏立项工作流

## 概述

本工作流覆盖从一句话创意到正式立项提案的完整路径。通过创意验证、核心概念定义、系统架构规划、可行性评估与立项决策五个阶段，确保项目在启动前经过充分论证，降低立项风险。

## P1/P2 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何会新增或改写项目文档的阶段，在产出完成后都必须执行 `skill: post-write-validator`，清空 Blocker 后方可进入下一阶段或提交审批。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若当前活跃文档发生切换，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审 / 评分文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 创意验证

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 1 周 |
| **参与 Agent** | @game-制作人（主导） |
| **所需 Skill** | `competitor-analysis` |
| **所需模板** | `TMPL_创意验证书` |

### 输入

- 一句话创意描述
- 目标平台（PC / Mobile / Console / 跨平台）
- 参考竞品列表（至少 3 款）

### 工作内容

1. **创意拆解** — @game-制作人 将一句话创意拆解为核心体验关键词、目标用户画像、差异化卖点
2. **竞品分析** — 调用 `competitor-analysis` 技能，对参考竞品进行多维度对比分析（玩法、商业模式、用户规模、口碑评价）
3. **市场需求评估** — 结合竞品分析结果，评估市场空间、目标用户量级、变现潜力
4. **创意验证书撰写** — 按 `TMPL_创意验证书` 模板输出结构化验证报告

### 输出

- `CONC_001_创意验证书.ai.md` — 创意可行性判定与依据
- `CONC_002_竞品分析报告.ai.md` — 竞品多维度对比矩阵
- `CONC_003_市场需求评估.ai.md` — 市场规模与机会分析

### 质量门禁

- 竞品分析覆盖至少 3 款同类产品
- 创意验证书包含明确的 GO / NO-GO 建议
- 市场需求评估包含数据支撑的用户规模估算

## Phase 2: 核心概念

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 2 周 |
| **参与 Agent** | @game-制作人（统筹）、@game-世界观（世界观方向）、@game-玩法策划（玩法定义） |
| **所需模板** | `TMPL_01_VISION`、`TMPL_02_PILLARS`、`TMPL_03_LOOPS` |

### 输入

- 创意验证书（Phase 1 产出）

### 工作内容

1. **愿景定义** — @game-制作人 主导，明确游戏的一句话愿景、情感目标、核心幻想
2. **设计支柱提炼** — @game-制作人 与 @game-世界观、@game-玩法策划 协作，从愿景中提炼 3-5 根设计支柱，每根支柱包含定义、体验目标与衡量标准
3. **核心循环设计** — @game-玩法策划 主导，定义 Moment-to-Moment / Session / Long-term 三层循环，标注正负反馈回路
4. **世界观概念草案** — @game-世界观 基于愿景与设计支柱，给出世界观初步方向（主题、调性、时代背景）

### 输出

- `VISION_001_愿景.ai.md` — 游戏愿景文档
- `PILLARS_001_设计支柱.ai.md` — 设计支柱定义文档
- `LOOP_001_核心循环设计.ai.md` — 核心循环设计文档

### 质量门禁

- 愿景文档包含一句话愿景、情感关键词、核心幻想
- 设计支柱不少于 3 根且各自包含衡量标准
- 核心循环覆盖三层时间尺度

## Phase 2b: 系统架构（与 Phase 2 并行）

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 与 Phase 2 并行 |
| **参与 Agent** | @game-系统架构师（主导） |
| **所需 Skill** | `mermaid-drawing` |
| **所需模板** | `TMPL_05_SYSTEMARCH` |

### 输入

- 核心循环文档（Phase 2 早期产出）
- 设计支柱文档（Phase 2 早期产出）

### 工作内容

1. **系统识别** — @game-系统架构师 从核心循环与设计支柱中识别关键子系统（战斗、经济、成长、社交等）
2. **系统拓扑绘制** — 调用 `mermaid-drawing` 技能，绘制系统间依赖关系拓扑图，标注数据流方向与耦合强度
3. **架构文档撰写** — 按 `TMPL_05_SYSTEMARCH` 模板输出系统架构文档，包含系统清单、分层结构、接口定义

### 输出

- `ARCH_001_系统架构总览.ai.md` — 系统架构设计文档
- 系统拓扑图（Mermaid 格式，嵌入文档）

### 质量门禁

- 系统拓扑图覆盖核心循环涉及的所有子系统
- 每个子系统有明确的职责边界与接口定义
- 系统间耦合度标注合理，无循环依赖

## Phase 3: 可行性评估

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 1 周 |
| **参与 Agent** | @game-技术总监（主导）、@game-制作人（商业视角） |

### 输入

- 核心概念文档（Phase 2 全部产出）
- 系统架构文档（Phase 2b 产出）

### 工作内容

1. **技术可行性分析** — @game-技术总监 评估核心玩法的技术实现难度、引擎选型建议、关键技术风险点
2. **资源评估** — 评估团队规模需求、工期估算、关键岗位需求
3. **风险矩阵构建** — 从技术、市场、团队、资金四个维度构建风险矩阵，标注概率与影响度
4. **商业可行性** — @game-制作人 评估预算需求、预期 ROI、商业模式可行性

### 输出

- `FEAS_001_技术可行性报告.ai.md` — 技术维度的可行性判定
- `RISK_001_风险矩阵.ai.md` — 四维度风险评估矩阵

### 质量门禁

- 技术可行性报告覆盖引擎、网络架构、性能、安全等维度
- 风险矩阵每项风险标注概率（高/中/低）与影响度（高/中/低）
- 高风险项均有对应缓解方案

## Phase 4: 立项决策

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 1 周 |
| **参与 Agent** | @game-制作人（最终裁决） |
| **所需 Skill** | `review-scoring` |
| **所需模板** | `TMPL_游戏概念书` |

### 输入

- 所有前序阶段产出文档
- 评分标准文档（如无，则使用项目默认评分标准）

### 工作内容

1. **文档汇总** — @game-制作人 汇总所有阶段产出，提炼关键结论
2. **立项提案撰写** — 按 `TMPL_游戏概念书` 模板，整合愿景、支柱、循环、架构、可行性分析为一份立项提案
3. **项目计划编制** — 制定项目整体计划，包含里程碑定义、阶段目标、关键交付物
4. **统一评分** — 调用 `review-scoring`，基于愿景清晰度、支柱成立度、核心循环完成度、可行性与风险控制给出总分与维度分
5. **立项评审** — @game-制作人 作为最终裁决者，结合评分报告给出 GO / CONDITIONAL-GO / NO-GO 决定

### 输出

- `PLAN_001_立项提案书.ai.md` — 完整的立项提案文档
- `PLAN_002_项目计划书.ai.md` — 项目整体计划与资源安排
- `PLAN_003_里程碑表.ai.md` — 关键里程碑与交付时间线
- `REVIEW_{NNN}_{YYYY-MM-DD}_立项评分报告.ai.md` — 统一评分结论与上下限说明

### 质量门禁

- 立项提案书覆盖愿景、支柱、循环、架构、风险五大板块
- 项目计划书包含至少 3 个里程碑节点
- 里程碑表每个节点有明确的交付物定义与验收标准
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 直接 NO-GO；6.0 - 6.9 最多 CONDITIONAL-GO；7.0+ 方可 GO
- 最终裁决结论明确，附决策依据
