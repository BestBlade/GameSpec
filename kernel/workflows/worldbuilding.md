# 世界观构建工作流

## 概述

本工作流覆盖从零开始构建游戏世界观的完整路径。从存在之基的哲学设定出发，逐步展开地理与势力、文化与叙事衔接，最终通过规范审查确保世界观的一致性与可扩展性。

## P1/P2 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L2 角色新建或修改项目文档后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可交接、提审或切换阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若某份文档成为当前活跃工作对象，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 存在之基

| 属性 | 值 |
|------|-----|
| **层级** | L0 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-世界观（主导）、@game-导航（启动分发） |
| **所需 Skill** | `naming-generation` |

### 输入

- 游戏愿景文档（Vision）
- 设计支柱文档（Pillars）
- 核心情感目标

### 工作内容

1. **本体论设定** — 定义世界的根本存在法则（物理规则、魔法/科技体系、生死观）
2. **时间轴构建** — 建立世界的纪元划分、关键历史节点、当前时代定位
3. **核心矛盾定义** — 确立驱动世界运转的核心矛盾（秩序vs混沌、科技vs自然等）
4. **命名体系** — 调用 `naming-generation` 技能，建立世界专属的命名规则与词汇表

### 输出

- 世界本体论设定文档
- 时间轴与纪元大事记
- 核心矛盾定义
- 命名规则与词汇表

### 质量门禁

- 本体论设定与游戏愿景一致
- 时间轴关键节点 ≥ 5 个
- 核心矛盾有明确的对立面与驱动力

## Phase 2: 地理与势力

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 5-7 天 |
| **参与 Agent** | @game-世界观（主导） |
| **所需 Skill** | `mermaid-drawing`、`naming-generation` |

### 输入

- 存在之基文档（Phase 1 产出）

### 工作内容

1. **地理体系构建** — 设计世界地图的宏观布局（大陆/区域/关键地标），标注地理特征与资源分布
2. **势力体系设计** — 定义主要势力/阵营（起源、理念、领地、关系），调用 `mermaid-drawing` 绘制势力关系图
3. **种族/族群设定** — 设计世界中的种族或族群（外貌、文化特征、能力倾向）
4. **势力动态** — 定义势力间的当前态势（同盟、敌对、中立）与潜在冲突点

### 输出

- 地理体系文档（含宏观地图描述）
- 势力体系文档（含势力关系图）
- 种族/族群设定文档

### 质量门禁

- 地理体系与本体论设定（如资源与魔法/科技体系）逻辑自洽
- 势力关系图无逻辑矛盾
- 每个势力有明确的动机与行为模式

## Phase 3: 文化与叙事衔接

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 5-7 天 |
| **参与 Agent** | @game-世界观（主导）、@game-叙事（叙事衔接） |
| **所需 Skill** | `naming-generation` |

### 输入

- 存在之基文档
- 地理与势力文档（Phase 2 产出）

### 工作内容

1. **文化体系设计** — 为各势力/区域设计独特文化（语言风格、信仰、习俗、禁忌）
2. **叙事钩子埋设** — @game-叙事 参与，在世界观设定中埋入叙事钩子（未解之谜、预言、历史悬案）
3. **玩家切入点定义** — 确定玩家在世界中的初始定位、身份背景与可探索的世界观入口
4. **术语体系完善** — 调用 `naming-generation` 完善世界特有术语、称谓、地名体系

### 输出

- 文化体系文档
- 叙事钩子清单
- 玩家切入点设定
- 完整术语词典

### 质量门禁

- 各文化体系与对应势力/地理特征一致
- 叙事钩子 ≥ 5 个，每个有展开潜力
- 术语词典无重名冲突

## Phase 4: 规范审查

| 属性 | 值 |
|------|-----|
| **层级** | L3 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-规范审查（主导） |
| **所需 Skill** | `document-validator`、`spec-standard-enforcer`、`review-scoring` |

### 输入

- 所有前序阶段产出文档
- 评分标准文档（如无，则使用项目默认评分标准）

### 审查模式适用

- 本阶段作为正式审查入口时，默认按 `full` 口径处理。
- `EXPL_*` / `CONC_*` 或显式声明 `review_mode: prototype` 的探索稿，不得直接作为 Phase 4 定稿输入。
- 日常补写可先按 `lean` 迭代，但进入正式定稿前必须补足一次 `full` 结论。

### 工作内容

1. **一致性校验** — 调用 `document-validator` 检查所有文档间的内容一致性（名称、设定、时间线无矛盾）
2. **规范合规校验** — 调用 `spec-standard-enforcer` 检查文档是否符合 GameSpec 标准格式与命名规范
3. **可扩展性评估** — 评估世界观设定的可扩展性，标注扩展受限的区域
4. **统一评分** — 调用 `review-scoring`，基于结构自洽、叙事承载力、可扩展性、可开发性给出总分与维度分
5. **审查报告** — 汇总所有问题，分级处理（阻塞/建议/信息）

### 输出

- 规范审查报告
- 世界观评审评分报告
- 通过 → 世界观文档正式定稿

### 质量门禁

- 全部文档间零内容矛盾
- 格式合规率 100%
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 或存在阻塞问题不得定稿；6.0 - 6.9 需修订后复审；7.0+ 方可直接定稿
- 阻塞问题清零方可定稿
