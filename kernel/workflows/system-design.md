# 系统设计工作流

## 概述

本工作流覆盖从 PRD/需求描述到开发交付的完整系统设计路径。经过需求分析、方案设计、设计评审、文档产出、逻辑验证、开发交付六个阶段，确保每一个游戏系统的设计经过严格论证与验证，最终产出可直接交付开发的高质量设计文档与配置表。

## P1/P2/P3 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L1 / L2 角色新建或修改项目文档、配置表说明文档或项目根 `active.md` 后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可进入下一阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若问题已收敛到 UE5 的实现边界、状态承载、SaveGame、Blueprint/C++ 分层或性能落点，可在 L2 起草后、L3 审查前按需调用 `@game-UE技术顾问`，输出 `UE 可落地性评估`；该节点不替代 `@game-技术总监` 的方向性裁决。
- 若 Phase 4 产出的正式设计文档成为当前活跃工作对象，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 需求分析

| 属性 | 值 |
|------|-----|
| **层级** | L0 |
| **周期** | 1-2 天 |
| **参与 Agent** | @game-导航（分发与路由）、@game-系统策划（边界确认） |

### 输入

- PRD / 需求描述文档
- 相关系统的现有设计文档（如有）

### 工作内容

1. **需求解析** — @game-导航 接收需求，识别所属系统域，分发给对应 Agent
2. **系统命名与边界确认** — @game-系统策划 明确系统的中英文名称、职责边界、不负责事项
3. **设计目标提炼** — 从需求中提炼 3-5 条可衡量的设计目标
4. **关联系统识别** — 列出与该系统有数据交互或逻辑依赖的其他系统
5. **复杂度评估** — 基于规则数量、状态数量、关联系统数评估系统复杂度（低/中/高）

### 输出

- 系统名称 & 职责边界定义
- 设计目标清单
- 关联系统列表（含交互方向）
- 复杂度评估结论

### 质量门禁

- 系统边界无歧义，不负责事项明确列出
- 设计目标可衡量、可验证
- 复杂度评估有量化依据

## Phase 2: 方案设计

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 3-6 天 |
| **参与 Agent** | @game-系统策划（主导）、@game-数值（可选，涉及数值子系统时参与） |
| **所需 Skill** | `mermaid-drawing`、`rule-writing` |
| **所需模板** | `TMPL_系统设计规范`、`TMPL_04_RESOURCEFLOW` |

### 输入

- 需求分析结果（Phase 1 产出）

### 工作内容

1. **概念模型构建** — @game-系统策划 定义系统核心实体、属性、关系，形成概念模型
2. **流程图绘制** — 调用 `mermaid-drawing` 技能，绘制系统核心流程图（状态机 / 时序图 / 流程图）
3. **原子规则集编写** — 调用 `rule-writing` 技能，将系统行为拆解为原子规则，每条规则包含前置条件、动作、后置效果
4. **资源流设计** — 按 `TMPL_04_RESOURCEFLOW` 模板，设计系统内外的资源（货币/道具/能量等）流转路径
5. **数值配置方案** — @game-数值 参与设计数值相关的配置结构与初始参数范围（如适用）
6. **异常处理设计** — 定义边界条件、异常情况的处理策略与降级方案

### 输出

- 系统设计文档（初稿）
- 核心流程图（Mermaid 格式）
- 原子规则集
- 数值配置方案（如适用）

### 质量门禁

- 概念模型覆盖所有核心实体与关系
- 每条原子规则包含前置条件、动作、后置效果三部分
- 流程图与原子规则集逻辑一致
- 异常处理覆盖率 ≥ 主流程分支的 80%

## Phase 3: 设计评审

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 1-2 天 |
| **参与 Agent** | @game-技术总监（技术可行性）、@game-系统架构师（架构合理性） |
| **所需 Skill** | `review-scoring` |

### 输入

- 设计文档初稿（Phase 2 产出）
- 核心流程图
- 原子规则集
- 评分标准文档（如无，则使用项目默认评分标准）

### 工作内容

1. **技术可行性评审** — @game-技术总监 评估设计方案的技术实现难度、性能风险、兼容性问题
2. **架构合理性评审** — @game-系统架构师 评估系统在整体架构中的定位、接口设计合理性、与其他系统的耦合度
3. **统一评分** — 调用 `review-scoring`，基于目标清晰度、规则完备性、架构合理性、技术可落地性给出总分与维度分
4. **评审意见汇总** — 汇总评审意见，分为「必须修改」「建议修改」「通过」三档
5. **评审决策** — L1 审批：通过 / 有条件通过（附修改要求）/ 打回重设计

### 输出

- 评审意见文档
- 设计评审评分报告
- 评审结论（通过 / 有条件通过 / 打回）

### 质量门禁

> **L1 审批通过方可进入下一阶段**

- 技术可行性无高风险阻塞项
- 架构合理性评分 ≥ 合格线
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 直接打回；6.0 - 6.9 最多有条件通过；7.0+ 且阻塞项清零方可通过
- 「必须修改」项为零时方可通过
- 打回时需附明确的修改方向与建议

## Phase 4: 文档产出

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 2-3 天 |
| **参与 Agent** | @game-系统策划（主导） |
| **所需 Skill** | `config-table-gen`、`mermaid-drawing` |
| **所需模板** | `TMPL_数值配置规范` |

### 输入

- 评审通过的设计方案（Phase 3 产出）
- 评审修改意见（如有条件通过）

### 工作内容

1. **设计文档定稿** — @game-系统策划 根据评审意见修订设计文档，产出正式版本（`.ai.md` 格式）
2. **配置表生成** — 调用 `config-table-gen` 技能，根据设计文档中的数值配置方案，生成结构化配置表
3. **UI 流程图绘制** — 调用 `mermaid-drawing` 技能，绘制用户交互流程图，标注每步的 UI 状态与转换条件
4. **文档元数据补全** — 补全文档 frontmatter（版本号、作者、审批状态、关联系统等）

### 输出

- 正式设计文档（`.ai.md` 格式）
- 配置表（结构化格式）
- UI 流程图

### 质量门禁

- 设计文档符合 `TMPL_系统设计规范` 格式要求
- 配置表字段定义完整，含类型、范围、默认值
- UI 流程图覆盖主流程与核心分支流程
- 文档 frontmatter 完整且格式正确

## Phase 5: 文档复核与定稿门禁

| 属性 | 值 |
|------|-----|
| **层级** | L3 |
| **周期** | 1 天 |
| **参与 Agent** | `workflow: document-review`（`@game-规范审查` → `@game-逻辑验证` → `@game-Spec架构师`） |
| **所需 Skill** | `document-validator`、`logic-review`、`spec-standard-enforcer`、`structure-diff`、`review-scoring` |

### 输入

- 正式设计文档（Phase 4 产出）
- 配置表

### 审查模式适用

- 本阶段接入 `workflow: document-review`，其 `review_mode` 分发规则以该工作流定义为准。
- 进入 Phase 5 的正式提交默认按 `full` 处理。
- `lean` 可用于局部修订的低成本复核，但若未走完完整回路，不得直接进入人工确认。
- `prototype` 仅适用于探索稿的最小结构检查，不得直接作为 Phase 5 的正式定稿输入。

### 工作内容

1. **规范审查** — 先进入 `@game-规范审查`，阻断 frontmatter、命名、格式和链接问题，并识别 `review_mode`
2. **逻辑验证** — 按 `review_mode` 分发规则决定是否进入 `@game-逻辑验证`，检查规则闭环、状态机和边界条件
3. **Spec 架构复核** — 按 `review_mode` 分发规则决定是否进入 `@game-Spec架构师`，检查模板、结构与跨文档引用质量
4. **汇总判定** — 汇总三层审查结果，明确通过 / 有条件通过 / 打回及回流阶段
5. **问题回流** — 若打回，默认回流至 Phase 2 或 Phase 4，由文档主责角色修订后重进本阶段

### 输出

- 规范审查报告
- 逻辑验证报告
- Spec 架构复核报告
- 统一审查总结论
- 通过 → 进入人工确认 → 最终产出 `.md` 正式文档

### 质量门禁

> **打回次数 ≥ 3 次时，自动升级至 L1 评审**

- 逻辑验证零矛盾、零死锁
- 规范校验通过率 100%
- Spec 架构复核零结构偏离、零断链
- 审查报告包含每条检查项的通过/不通过状态与回流指令

## Phase 6: 开发交付

| 属性 | 值 |
|------|-----|
| **层级** | L1 |
| **周期** | 按排期 |
| **参与 Agent** | @game-技术总监（主导）、@game-系统策划（答疑支持） |

### 输入

- 通过逻辑验证的正式设计文档
- 配置表

### 工作内容

1. **技术实现方案制定** — @game-技术总监 基于设计文档，制定技术实现方案（模块拆分、接口定义、实现优先级）
2. **配置表数据填充** — 根据技术方案确认最终配置表数据
3. **开发答疑** — @game-系统策划 在开发过程中提供设计意图说明与细节答疑
4. **验收测试** — 开发完成后，按设计文档中的验收标准进行功能验收

### 输出

- 技术实现方案
- 最终配置表数据
- 验收报告

### 质量门禁

- 技术实现方案覆盖设计文档中所有功能点
- 验收测试通过率 ≥ 95%
- 验收报告包含每项功能的测试结果
