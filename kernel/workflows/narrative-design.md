# 叙事设计工作流

## 概述

本工作流覆盖从叙事大纲到对白脚本的完整叙事设计路径。通过顶层定调、核心三角构建、结构与节拍编排、交付与审查四个阶段，确保游戏叙事与世界观、玩法的深度融合，最终产出可直接用于开发的叙事脚本。

## P1/P2 接线协议

- 已有项目进入本工作流前，应先由 `@game-导航` 执行 `skill: session-start`，恢复 `active.md` / `.gamespec-state.yaml` 的当前工作锚点。
- 本工作流内任何 L2 角色新建或修改项目文档后，必须先执行 `skill: post-write-validator`，清空 Blocker 后方可交接、提审或进入下一阶段。
- 在长会话压缩、任务中断、跨角色交接或准备上交 L1 前，当前执行角色必须执行 `skill: session-compact`，将 `Blockers` / `Pending Handoffs` 与 session 元数据写回项目检查点；handoff 不自动改写 `current_agent`。
- 若当前活跃文档发生切换，应同步更新项目根 `active.md`，并把该文件纳入同一次写后校验。
- P4 事实源语义同样生效：`prototype` 不得充当稳定依赖；`full` / `lean` 若依赖 `lean` 文档，正文前部必须显式写明当前口径或工作假设；`lean` 文档不得直接进入 `.ai.md -> .md`。

## M1 意图前置步骤（所有阶段通用）

本工作流的任何阶段在设计产出前，均须执行以下前置步骤：

1. L0 (`@game-导航`) 将用户原始意图（原话）传递给目标 L2。
2. L2 使用 `skill: intent-spec` 和 `TMPL_DESIGN_INTENT` 产出 `INTT_{PHASE}_{名称}.ai.md`。
3. L1 (`@game-制作人` 或 `@game-系统架构师`) 完成三确认：意图清晰性、翻译忠实性、边界合理性。
4. 三确认未全部通过 → 打回修改 INTT_；全部通过 → 阶段设计工作开始。
5. 设计过程中意图发生变更 → 停止当前阶段，更新 INTT_，重新 L1 确认。

## 文档身份与引用硬约束

- 活文档文件名必须满足：`{system_id}_{标题摘要}.ai.md`
- 快照 / 评审文档文件名必须满足：`{system_id}_{YYYY-MM-DD}_{标题摘要}.ai.md`
- 文件名前缀必须与 frontmatter 中的 `system_id` 完全一致
- `dependencies` 只填写 `system_id`
- `dependencies` 非空时，正文前部必须提供与其一一对应的“依赖文档 / 相关文档”链接列表
- `scope` 只写范围摘要；若范围涉及具体项目文档，正文必须提供可跳转的范围说明或清单
- 正文凡出现其他项目文档引用（包括范围说明、摘要、表格、附录、证据列表、行动项），都必须使用 Markdown 相对路径链接；首次引用至少包含 `system_id` 与标题摘要，评审证据引用应尽量带行号

## Phase 1: 顶层定调

| 属性 | 值 |
|------|-----|
| **层级** | L0 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-叙事（主导）、@game-世界观（世界观对齐）、@game-导航（启动分发） |

### 输入

- 游戏愿景文档
- 世界观设定文档
- 设计支柱文档

### 工作内容

1. **叙事主题定义** — 确立游戏核心叙事主题（如救赎、探索、生存、复仇等）
2. **叙事调性确认** — 定义叙事调性（史诗/日常、严肃/轻松、写实/夸张）
3. **叙事结构选型** — 选择宏观叙事结构（线性/分支/开放/碎片化）
4. **世界观叙事锚点** — @game-世界观 确认叙事可使用的世界观元素与禁区

### 输出

- 叙事主题与调性定义文档
- 叙事结构选型方案
- 世界观叙事约束清单

### 质量门禁

- 叙事主题与游戏愿景一致
- 调性定义包含正面示例与反面示例
- 叙事结构选型有设计理由支撑

## Phase 2: 核心三角

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 5-7 天 |
| **参与 Agent** | @game-叙事（主导）、@game-世界观（设定校验） |
| **所需 Skill** | `narrative-spec` |

### 输入

- 顶层定调文档（Phase 1 产出）
- 世界观文档（势力、文化、叙事钩子）

### 工作内容

1. **主角设计** — 定义主角的身份、动机、内在冲突、成长弧线
2. **核心角色群像** — 设计核心 NPC 群像（同伴、对手、导师），标注与主角的关系动态
3. **核心冲突链** — 构建驱动叙事的核心冲突链（外在冲突 + 内在冲突），定义冲突升级节奏
4. **叙事规范产出** — 调用 `narrative-spec` 技能，按规范格式输出角色档案与冲突链文档

### 输出

- 主角设计文档
- 核心角色群像文档
- 核心冲突链文档

### 质量门禁

- 主角有明确的内在冲突与成长弧线
- 核心 NPC ≥ 3 个，各自有独立动机
- 冲突链有清晰的升级节奏与高潮预设

## Phase 3: 结构与节拍

| 属性 | 值 |
|------|-----|
| **层级** | L2 |
| **周期** | 5-7 天 |
| **参与 Agent** | @game-叙事（主导）、@game-玩法策划（玩法节奏对齐） |
| **所需 Skill** | `quest-spec`、`copywriting-polish` |

### 输入

- 核心三角文档（Phase 2 产出）
- 核心循环定义（来自玩法策划）

### 工作内容

1. **章节/幕结构编排** — 按选定叙事结构，划分章节或幕，定义每幕的叙事目标与情感曲线
2. **节拍表设计** — 在每幕内设计叙事节拍（beat），标注事件、情感、玩法关联
3. **任务/剧情脚本** — 调用 `quest-spec` 技能，将节拍转化为具体的任务设计或剧情脚本
4. **对白初稿** — 编写关键场景对白初稿，调用 `copywriting-polish` 进行文案润色
5. **玩法节奏对齐** — @game-玩法策划 确认叙事节奏与玩法节奏的匹配度

### 输出

- 章节/幕结构文档
- 节拍表
- 任务/剧情脚本
- 关键对白文稿

### 质量门禁

- 每幕有明确的叙事目标与情感高低点
- 节拍表与玩法节奏无冲突
- 对白符合角色人设与叙事调性

## Phase 4: 交付与审查

| 属性 | 值 |
|------|-----|
| **层级** | L2 → L3 |
| **周期** | 3-5 天 |
| **参与 Agent** | @game-叙事（文档整理） + `workflow: document-review`（`@game-规范审查` → `@game-逻辑验证` → `@game-Spec架构师`） |
| **所需 Skill** | `copywriting-polish`、`document-validator`、`logic-review`、`spec-standard-enforcer`、`structure-diff`、`review-scoring` |

### 输入

- 所有前序阶段产出文档
- 任务/剧情脚本与对白文稿
- 评分标准文档（如无，则使用项目默认评分标准）

### 审查模式适用

- 本阶段接入 `workflow: document-review`，其 `review_mode` 分发规则以该工作流定义为准。
- 进入交付与审查的正式提交默认按 `full` 处理。
- `lean` 可用于局部补写或结构修订的低成本复核，但若未走完完整回路，不得直接进入正式交付。
- `prototype` 仅适用于探索性叙事稿的最小结构检查，不得直接作为本阶段定稿输入。

### 工作内容

1. **文档整合** — @game-叙事 将所有叙事文档整合为可交付的完整叙事设计包
2. **文案终审** — 调用 `copywriting-polish` 对所有对白与叙事文本进行最终润色
3. **统一文档审查** — 启动 `workflow: document-review`，按 `review_mode` 执行 full / lean / prototype 分流复核
4. **统一评分** — 调用 `review-scoring`，基于主题-角色-结构同轴度、节拍完成度、文本质量、可交付性给出总分与维度分
5. **交付确认** — 结合三层审查与评分结果，确认叙事文档是否可正式定稿交付

### 输出

- 完整叙事设计包（定稿）
- 规范审查报告
- 逻辑验证报告
- Spec 架构复核报告
- 叙事评审评分报告
- 交付确认记录

### 质量门禁

- 叙事设计包结构完整（主题→角色→结构→脚本→对白）
- 规范审查通过率 100%
- 逻辑验证零因果断裂、零未定义分支
- Spec 架构复核零结构偏离、零断链
- 与世界观设定零矛盾
- 对白文本无错别字、语病
- 必须产出“总分 / 维度分 / 为什么不是更低 / 为什么不是更高”
- 总分 < 6.0 或存在阻塞问题不得定稿；6.0 - 6.9 需修订后复审；7.0+ 方可直接定稿
