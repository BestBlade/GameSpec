# P0 定稿：会话状态与审查模式

## 决策摘要

- 实际在用的体系是 `gamespec`，不是 `cop-gamespec`。
- P0 的两项内容应一起定义，但书写顺序应先定 `review_mode`，再定 `active.md`，因为 `active.md` 需要显式记录当前审查模式。
- 本文件即为当前 P0 冻结口径，后续 P1 实施以此为准。
- P0 的落地范围是“定义规则并接入现有宪法/审查工作流”，不在本阶段引入新的命令系统或完整自动化。

---

## 1. 审查强度模式定义

### 1.1 目标

在不破坏当前 GameSpec 核心纪律的前提下，为不同设计阶段提供不同强度的审查路径：

- 保留正式设计文档的高强度质量门禁。
- 允许日常迭代降低深度审查成本。
- 为探索稿提供最小但不为零的结构约束。

### 1.2 不变约束

无论处于哪一种 `review_mode`，以下规则始终不变：

- `.ai.md -> .md` 的 HITL 生命周期不变。
- L0 不写、L1 不写、L3 不改的权限边界不变。
- `system_id`、命名、Frontmatter、依赖引用的基础规范不失效。
- 若文档中出现数值规则或公式，仍必须遵守 `{{VAR_*}}` 数值抽象规则。
- 任一模式下出现断链、身份冲突、非法命名、缺失必填元数据，均视为阻塞问题。

### 1.3 模式总览

| 模式 | 适用场景 | 核心语义 | 能否直接进入正式定稿链 |
|------|----------|----------|------------------------|
| `full` | 正式设计、跨系统变更、正式复提、进入审查前 | 当前完整 L3 强度 | 可以 |
| `lean` | 日常迭代、单文档修订、局部补全 | 保留硬校验，按条件触发深度审查 | 可以，但进入正式定稿前需补足 `full` |
| `prototype` | 探索稿、创意验证稿、快速验证 | 仅做最小结构校验，不做深度逻辑审查 | 不可以 |

### 1.4 工作流默认解释

为避免 `review_mode` 缺失导致执行口径漂移，P0 冻结以下默认解释：

- 文档 frontmatter 显式声明 `review_mode` 时，以显式声明为准。
- `EXPL_*` / `CONC_*` 草稿在未显式声明时，可按 `prototype` 处理。
- 任意文档一旦进入 `workflow: document-review` 且未显式声明 `review_mode`，默认按 `full` 处理。
- 其他日常活文档在未显式声明且未进入正式审查流时，按 `lean` 处理。

这一定义只解决执行默认值，不改变作者主动升级或降级审查强度的权利。

### 1.5 三档模式的检查边界

| 检查层 | `full` | `lean` | `prototype` |
|------|--------|--------|-------------|
| 文档身份与命名 | 必查 | 必查 | 必查 |
| Frontmatter 必填字段 | 必查 | 必查 | 必查 |
| `system_id` 唯一性 | 必查 | 必查 | 必查 |
| 基础链接有效性 | 必查 | 必查 | 必查 |
| 模板章节完整性 | 必查 | 仅对核心活文档与正式复提文档必查 | 只检查是否存在最小骨架 |
| 规则一致性 / 边界条件 / 死锁 | 必查 | 仅在触发条件满足时检查 | 不查 |
| 跨文档一致性 / Spec 架构复核 | 必查 | 仅在触发条件满足时检查 | 不查 |
| 评分与回流指令 | 必查 | 仅对进入正式链的文档执行 | 不执行 |

### 1.6 `lean` 模式的深度审查触发条件

处于 `lean` 模式时，以下任一条件满足，仍应升级到完整的深度审查路径：

- 文档属于核心活文档，而不是探索稿或一次性快照。
- 本次修改影响多个系统、多个角色或多个依赖文档。
- 文档即将进入 `review` 状态或准备执行正式复提。
- L1 明确要求补做完整逻辑验证或 Spec 架构复核。
- 本轮修改包含新的规则集、状态机、接口定义、公式或资源流变更。

### 1.7 `prototype` 模式的适用边界

`prototype` 不是免审模式，而是受限模式。其约束如下：

- 默认仅适用于 `EXPL_*`、`CONC_*` 这类探索性、验证性、尚未进入正式设计链的文档。
- 若其他前缀文档需要使用 `prototype`，必须在 Frontmatter 中显式声明，例如：

```yaml
review_mode: prototype
prototype_reason: 该文档仅用于快速验证，不进入正式依赖链
```

- `prototype` 文档不得直接作为正式系统设计、正式叙事设计、正式关卡设计、正式数值设计的最终提交版本。
- `prototype` 文档不得直接进入正式定稿链。
- 若后续需要被正式文档引用，必须先升级到 `lean` 或 `full` 并完成对应审查。

### 1.8 `prototype` 文档依赖隔离

为避免低强度探索稿污染正式设计链，增加以下依赖规则：

- `full` 文档不得依赖 `prototype` 文档。
- `lean` 文档不得把 `prototype` 文档作为稳定前置依赖。
- `prototype` 文档可以依赖 `full` / `lean` 文档，但必须明确自己是探索性派生，不反向要求上游为其背书。
- 若某份 `prototype` 文档被多个正式文档引用，则它已经不再是探索稿，应升级为 `lean` 或 `full`。

### 1.9 模式切换规则

| 切换 | 是否允许 | 说明 |
|------|----------|------|
| `prototype -> lean` | 允许 | 探索稿开始进入可复用设计阶段 |
| `prototype -> full` | 允许 | 直接进入正式提审前的全面校验 |
| `lean -> full` | 允许 | 正式进入定稿链前的标准升级 |
| `full -> lean` | 允许 | 仅用于退回日常迭代，不改变已记录的审查结论 |
| `full/lean -> prototype` | 谨慎 | 仅用于明确回退为探索稿，不得保留“可直接定稿”预期 |

模式切换后，必须同步更新：

- 当前项目的 `active.md`
- 必要时更新项目根的 `.gamespec-state.yaml` 备注或当前阶段说明

### 1.10 默认推荐

| 场景 | 推荐模式 |
|------|----------|
| 创意探索、概念试写、快速问题拆解 | `prototype` |
| 日常补写、局部迭代、单文档修订 | `lean` |
| 正式评审、跨系统变更、归档前复提 | `full` |

## 2. `active.md` 会话状态规范

### 2.1 目标

为当前会话提供一个可恢复的、轻量级的工作锚点，用来记录：

- 当前正在处理哪一份文档
- 当前处理到哪一节
- 当前使用哪一种审查模式
- 最近做了哪些关键决策
- 下一步应该做什么

它不是项目级真相源，而是会话级检查点。

### 2.2 文件位置

建议位置：

`gamespec/projects/{项目名}/active.md`

选择项目根而不是额外引入 `production/session-state/active.md` 的原因：

- 当前 `gamespec` 项目目录已经形成稳定使用习惯，项目根是最直接的工作锚点。
- `active.md` 与当前项目文档关系最紧密，放在项目根更直观。
- P0 不应为此额外搬入外部工作室模板的完整目录结构。

### 2.3 版本控制规则

- `active.md` 是会话级检查点，不是长期设计资产。
- `active.md` 不应纳入版本控制。
- 会话恢复依赖当前本地状态，而不是 Git 历史。

### 2.4 与 `.gamespec-state.yaml` 的分工

| 文件 | 层级 | 作用 |
|------|------|------|
| `.gamespec-state.yaml` | 项目级 | 记录工作流阶段、全局阻塞、里程碑、长期备注 |
| `active.md` | 会话级 | 记录本次会话的当前文档、当前章节、当前模式、下一步动作 |

约束：

- `.gamespec-state.yaml` 是长期状态，更新频率较低。
- `active.md` 是工作检查点，允许高频覆盖更新。
- 会话恢复时先读 `active.md`，再读相关文档和 `.gamespec-state.yaml`。

### 2.5 最小字段

`active.md` 至少应包含以下字段：

```yaml
---
project: <project-id>
current_workflow: worldbuilding
current_agent: @game-世界观
current_document: WORLD_002_样本城_工房镇.ai.md
current_section: "3.2 经济结构"
review_mode: lean
updated: 2026-04-20 20:30
---
```

正文至少应包含以下三个区块：

- `## Recent Decisions`
- `## Next Step`
- `## Open Questions`

### 2.6 推荐扩展字段

如需更强的恢复能力，可增加以下字段或区块：

- `touched_files`：本次会话涉及的文档列表
- `pending_handoffs`：待交接给其他 L2 或 L1 的事项
- `blockers`：当前阻塞点
- `evidence`：最近一次审查报告或关键证据链接

### 2.7 推荐模板

```markdown
---
project: <project-id>
current_workflow: worldbuilding
current_agent: @game-世界观
current_document: WORLD_002_样本城_工房镇.ai.md
current_section: "3.2 经济结构"
review_mode: lean
updated: 2026-04-20 20:30
---

## Recent Decisions
- 工房镇主产业确定为武器锻造，不再保留药剂工坊为主产业。
- 镇内三层社会结构改为“工坊主 / 学徒 / 外来行商”而非“贵族 / 平民”二元结构。

## Next Step
- 完成 `3.3 社会结构`。
- 完成后执行一次基础格式检查，再决定是否升级到 `full` 提审。

## Open Questions
- 是否需要把黑市锻造链条前置到本镇，还是留到下一座城市展开？

## Pending Handoffs
- 若经济结构定稿，需向 `@game-数值` 交接资源产出入口与变量语义。
```

### 2.8 更新时机

以下时点必须更新 `active.md`：

- 开始处理一份新的文档时
- 完成一个章节或一个明确决策时
- `review_mode` 发生变化时
- 收到 L1 裁决或 L3 审查结论时
- L2 之间发生交接时
- 会话准备压缩、结束或中断前

### 2.9 使用规则

- `active.md` 只记录当前最活跃的一条工作流，不做会话历史堆积。
- 历史记录应进入正式文档、审查报告或 `.gamespec-state.yaml`，而不是无限增长地堆在 `active.md` 中。
- 若当前会话已切换主题，应先清理旧状态，再写入新的当前状态。

---

## 3. P1 对接建议

当 P0 口径稳定后，P1 可以按以下顺序接入：

1. 先做 `post-write validator`，校验 `review_mode`、`current_document`、Frontmatter 最小字段、原型稿依赖隔离等基础规则。
2. 再做 `session-start`，启动时优先读取 `active.md` 与 `.gamespec-state.yaml`，恢复当前工作锚点。
3. 若 `active.md` 的 `updated` 字段距离当前时间超过 48 小时，`session-start` 应发出 stale warning，而不是静默恢复旧上下文。
4. 在上述两项稳定后，再考虑状态汇总、自动 compact、交接摘要等扩展能力。

---

## 4. 当前建议

- 排序总体合理。
- 仅建议把 P0 内部顺序固定为：先定义审查模式，再定义 `active.md`。
- P1 中若只能先做两项自动化，优先：`post-write validator` 与 `session-start`。
- `REQ-ID`、迁移审计、UE 专家扩展仍应保留在后续阶段，不建议前置到 P0。

---

## 5. P0 完成定义

满足以下条件，即视为 P0 完成：

- `review_mode` 的三档语义、默认解释、升级/降级规则被明确写定。
- `prototype` 的前缀边界、显式例外与依赖隔离规则被明确写定。
- `active.md` 的位置、字段、使用时机与“不入版本控制”规则被明确写定。
- 全局宪法与正式审查工作流已引用或吸收这些口径。
- P1 实施范围已被单独文档化，后续自动化不再需要重新讨论 P0 原则。
