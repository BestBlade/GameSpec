## 概述

`intent-spec` 是 M1（意图先于产出）的核心执行技能。

它的职责不是替用户做决定，而是**将用户已经表达的方向翻译成可检查、可复核的条款**。翻译过程必须透明——用户原始意图和结构化翻译永远同框，确保 L1 能同时校验两者的忠实性。

## 何时调用

以下情况必须调用本技能：

- `/gamespec:propose` 开始一个新工作流阶段前
- `/gamespec:apply` 的产出目标尚未有对应的 INTT_ 文档时
- L1 打回已有 INTT_ 要求重新翻译时
- 设计过程中发现意图需要变更（漂移检测触发）

## 步骤

1. **接收原始意图** — 读取 L0 传递的用户原话。此文本不得改写、不得概括、不得"润色"。
2. **加载上下文** — 读取 `.gamespec-state.yaml` 获取当前阶段、driver/parked/quarantined 权限、已解锁变量列表。
3. **翻译为结构化条款**:
   - 预期成果：明确到文档级别（哪个 system_id 的什么文档）
   - 设计目标：用可验证的条款描述设计应达成什么（而非怎么做）
   - 翻译说明：记录从原始意图到条款的推理链——哪些地方做了澄清、哪些地方保留了模糊
4. **划定边界** — 明确不在范围内的事项及其排除原因。引用关联但不纳入本阶段的系统/概念。
5. **定义完成信号** — 每条信号必须可被 L3 或 L1 验证。信号应对齐模板的 `section_signals`（如已定义）。
6. **预注册停止条件** — 什么会触发暂停/回退/重开意图。默认包含：发现与已有设定的冲突、范围蔓延超出边界、完成信号在当前阶段无法验证。
7. **生成 INTT_ 文档** — 使用 `TMPL_DESIGN_INTENT` 模板，以 `.ai.md` 后缀输出到项目对应阶段目录。
8. **提交 L1 确认** — 将 INTT_ 文档提交给 L1（制作人或系统架构师）进行三确认：意图清晰性、翻译忠实性、边界合理性。

## 约束

- INTT_ 文档使用 `.ai.md` 后缀（与所有 GameSpec 草稿一致）
- INTT_ 的 `system_id` 前缀为 `INTT`，文件名格式：`INTT_{PHASE}_{名称}.ai.md`
- L2 不得在没有 L1 确认的情况下开始设计产出
- L1 打回 INTT_ 时，L2 必须修改后重新提审，不得跳过
