# Core Concepts

GameSpec exists to protect game design truth during long AI-assisted work.

## Creative States

```text
Spark -> Thread -> Candidate -> Canon
```

| State | Purpose | Governance |
| --- | --- | --- |
| Spark | Preserve a raw idea. | Capture only. |
| Thread | Explore a line of thought. | Light trace. |
| Candidate | Prepare a consequential change for review. | Evidence required. |
| Canon | Record accepted project truth. | Strict review and decision trail. |

This is not a quality ladder. A failed Thread can be valuable. A Spark can stay a Spark forever. A Candidate can be rejected without being deleted.

## Project Truth

Project truth is the set of facts future work may trust: vision, pillars, systems, narrative, worldbuilding, levels, reviews, active state, decisions, archives, and implementation handoffs.

GameSpec treats project truth as project-owned. Installers and package updates must not overwrite it as part of a product refresh.

## Overlay

An overlay contains judging principles: producer taste, genre doctrine, constraints, or project-specific rules.

Overlay is optional. Some projects need explicit philosophy. Some do not. An overlay can influence judgment, but it does not become canon unless the project accepts it.

## Admission Review

Admission Review is a low-frequency commitment gate. It is useful when:

- a new idea might become a formal project
- an existing project needs a serious re-greenlight check
- a major pivot changes the investment thesis

Admission Review is not brainstorming. It consumes creative material and project truth as evidence, then prepares a human-owned GO, CONDITIONAL-GO, or NO-GO decision.

## Implementation Handoff

Implementation should consume reviewed design intent, not re-invent it.

A strong handoff names:

- accepted design truth
- unresolved risks
- dependencies and affected documents
- acceptance criteria
- evidence that the design is ready for downstream work
